package: patterns.cea

Invocation of init on a CEA service

The purpose of the init operation is to create an Application object inside the service that represents a job-step. This Application contains in its state the details of the "tool" (i.e. the business logic) to be run plus the parameters for a single run of the tool. The Application is triggered by a subsequent call to the excute operation of the service.

The formal parameters of the job-step are defined in the service's library of application descriptions. The actual arguments are passed in by the client as an object of type Tool (see diagram for qualified class-name) which is used in the service to generate the Application from the ApplicationDescription. The Tool is converted to a _tool by the delegate and back to a Tool by the service skeleton: this works round serialization problems in Axis and Castor.

At the end of this collaboration, the service makes the Application persistent, such that it is available for subsequent execute operations. The service returns an identifier for the Application to the client.

Note that ExecutionController, ApplicationDescriptionLibrary, ApplicationDescription and Application are interfaces while the other entities are classes. A sub-system implementing CEA would reuse the classes shown here and provide its own implementation of the interfaces.
