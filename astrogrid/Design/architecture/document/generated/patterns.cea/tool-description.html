package: patterns.cea

Static representation of an application in CEA

The classes shown here run inside a CEA-compliant service.

In CEA, Application is an interface to an object that wraps some business logic. The logic may be a local Java class, an executable to run in a sub-process, or an on-line service; the implementation of Application handles the details. The parameters for the business logic are set when the Application is created, so the Application represents a job-step rather than a reusable tool.

Job-steps in CEA are executed asynchronously. The execute() method on Application starts the job-step and returns immediately. To find out whether the step completed or failed, the caller must set an Observer-pattern object to watch the job-step; the addObserver method on Application supports this. Because the parameters of the job-step are part of the Application's state, the Application has a simple execute() method, without parameters, to start the job-step.

Services using CEA have an ApplicationDescriptionLibrary of possible applications, each described by an ApplicationDescription which contains one or more ApplicationInterfaces. The ApplicationDescriptions are factories for the Applications. Once a Service has got an Application out of an ApplicationDescription, it can invoke it trivially, since the details of the job-step are part of the application's state.

All the CEA parts here are interfaces that must be implemented. Application is the primary interface that must be implemented in a sub-system.
