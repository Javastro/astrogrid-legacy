Activity Choose authority ID

The registry "owns" an authority ID called the "managed authority ID". A registry may only accept resource registrations under its managed ID and no two registries may use the same managed ID.

In the current system, a registry may have only one such (later systems may allow multiple managed IDs). All resources registered in this registry will take the managed ID and their IVORNs will differ only in the resource-key part.

The managed ID must be unique throughout the IVO, not just within AstroGrid. It may help to base it on the internet domain in which the registry runs.

One benefit to a site of running its own publishing registry is that it gets to choose a custom authority ID. Since authority IDs are visible to end users, this may act as a form of trade-mark.

Activity Choose full or publishing registry

"Publishing" registries take in registrations of resources and make them available to other registries. "Full" registries "harvest" (i.e. copy and collect) registrations from publishing registries to produce a complete view of all resource in the IVO. In a mature VO, all full registries know about all publishing registries and thus are equivalent. Software that looks up registrations engages with a full registry.

The registry software from AstroGrid can work as a publishing registry, as full registry, or as a hybrid registry with both roles.

An AstroGrid site site does not need a local publishing registry to publish its services - it can use any publish registry within AstroGrid - but it may be more convenient for the site operators to run their own publishing registry.

A site might choose to run a full registry in order to have a local, low-latency service of the resource metadata.

Activity Set basic configuration

Sysadmin edits the properties file astrogrid.properties that is carried inside the deployed copy of astrogrid-registry. The important properties are:

org.astrogrid.registry.authorityId
The managed authority ID for this registry.
registry.harvest.*
Controls for harvesting of other registries by this registry. Set registry.harvest.enabled to false for a publishing reistry that does not itself harvest.

There are a number of other properties concerned with internal operation of the web-application. Please see the installation guide for details.

Activity Configure harvesting by other registries

Sysadmin copies the registry's self-registration - the VOResource for the registry itself - and submits it as a registration to each external full-registry of interest. If the external registry is an Astrogrid registry, then Sysadmin uses the web interface built into the full registries web-application. This is a web form into which the a VOResource can be pasted.

Most full registries are hybrids that act as publishing registires to each other. Thus, registering the new registry in one external full-registry may be enough to make the new registry known throughout the IVO. Alternatively, there may be clumps of registries with connections inside a clump (e.g. within a national VO project) but not between the clumps. Sysadmin work out the current registry topology in the IVO and register in enough full registries to get the desired coverage.

Full registry will now harvest registrations from the new, local registry at some point in the future. Sysadmin has no control over when this happens.

Activity Configure harvesting of other registries

Sysadmin makes a list of all publishing registries of interest. For each such registry, sysadmin obtains the self-registration (VOResource) for that registry. The easiest way to do this is to query a full registry for all its registrations of type RegistryType; it may be necessary to query several such full registries in different parts of the IVO to find all the publishing registries.

Sysadmin makes a VODescription document containing the VOResources for each publishing registry. Sysamin submits this document as a registration in the new, local registry by pasting it into a form of the web interface. The local registry then records each VOResource in the VODescription as a new registration.

The local registrywill now initiate the use-case "harvest resources" (q.v.) at the intervals that were set by Sysadmin in its basic, internal configuration.

Activity Register self

The registry must be configured with additional metadata to describe it. These metadata are public (c.f. the basic configuration which is private) and are available via the registry's web and web-service interfaces. The self-registration is entered as the first resource to be registered in the new registry.

Sysadmin creates an XML document containing a VOResource of type RegistryType. This document includes:

  • The managed authority ID for the registry.
  • A title and description for this registry.
  • Contact details for the registry operator.
  • The URL for the registry's harvesting (OAI) interface.

Note that including the URL for the harvesting interface does not automatically cause the registry's contents to be harvested. This URL has to be set even if the registry is not to be harvested at all.
