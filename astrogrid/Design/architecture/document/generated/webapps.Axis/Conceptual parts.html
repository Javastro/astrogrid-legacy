Conceptual parts of an Axis web application

An Axis web-service is part of the configuration of an Axis servlet running in a web application on a servlet engine. At run-time, the service has no separate existence. If the servlet engine, web application or servlet fail or are withdrawn, then the service endpoint ceases to exist.

AstroGrid service sub-systems are installed by deploying their web applications into a pre-installed servlet engine. AstroGrid sub-systems are upgraded by deleting their web applications and deploying new versions of the web applications.

Once deployed, the important parts of a web service are its implementation class (and the classes called by the implementation class, which are not drawn here) plus the WSDL contract. See the service interfaces diagram for details of the contract. A given service may also have configuration files and data files, but these are not part of the common pattern and must be separately designed for each service.

The Axis servlet contains an engine made of classes supplied with Axis. This engine controls access to the service-implementation classes.

The set of services run in the servlet is controlled by a deployment descriptor: an Axis-specific XML document. The descriptor also controls how the run-time WDL contract is derived.

The web application's servlet descriptor is an XML document written to the J2EE servlet standard. It determines the set of servlet running in the web application and details of their context paths.

Note that the servlet engine may be running any number of web applications based on Axis, and also web applications that dont use Axis.

The servlet engine provides exactly one set of configuration properties to all the web services. This configuration is an AstroGrid feature that is common to all AstroGrid web applications. The intention is that system integrators deploying AstroGrid sub-systems can keep a configuration outside any web application. Thus, sub-systems can be updated without losing the configuration.
