Component Diagram Physical parts
package: webapps.Axis

Physical parts of an Axis service and its server

The server infrastructure must be installed first on the host computer. The web application is then deployed into the server infrastructure.

AstroGrid services are only supported on Jakarta-Tomcat 5. They would probably work in Jakarta-Tomcat 4; earlier versions of the services ran with this engine, but current versions may have problems and back-porting is not intended. No attempt has been made to run the services in any other engine.

The common configuration must be set up in the server infrastructure. Several methods are possible, but the normal method is to write the properties into a file (as name-value pairs, in the usual way for properties of Java programmes) and to put that file in the /conf sub-directory of the Jakarta-Tomcat installation. The location of the properties file is then advertized to the web applications by setting the property org.astrogrid.config in Tomcat's JNDI engine. Web applications consume the properties file using library code in the org.astrogrid.commons.config package (not shown in this diagram.

The web application itself contains the Axis engine in axis.jar, plus subordinate jars (not shown), and the service implementation in one jar plus subordinate jars (not shown). There will typically be 3rd-party jars which are not supplied with Axis or produced by AstroGrid. All the jars live in the lib directory of the web application. The service implementation and its supporting classes may, in some services, be stored as loose .class files in the classes directory of the web application. This is an alternative to packaging them in a jar, and the choice depends on the build system for the web application, not the process of deploying the web application into the server.

The Axis servlet is executable only inside the servlet engine.

The web application may contain the WSDL contract as a .wsdl file which the Axis engine reads in and adjusts. However, most AstroGrid sub-systems set the Axis engine to generate the contract on demand from the compiled implementation-class. Some of these sub-systems include a .wsdl file which is not used at run time; others omit the .wsdl file.
