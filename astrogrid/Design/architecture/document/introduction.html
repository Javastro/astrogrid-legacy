Introduction

About AstroGrid

AstroGrid is the UK's virtual-observatory project. AstroGrid is funded (by the Particle Physics and Astronomy Research Council) to produce software within which data archives and data-processing software can be used. AstroGrid is not itself a producer of data-processing software for the UK (other projects, notably Starlink, do that), nor is it an operating organization for British parts of the virtual observatory (data-centres at British universities do that). AstroGrid's product is a software framework that can be deployed by other organizations. This document describes that product.

More information about AstroGrid as an organization is available on AstroGrid's main web-site.

About the virtual observatory

The virtual-observatory (VO) movement started with the aim of unifying all the existing on-line archives. During the long gestation of the VO, the grid paradigm of distributed processing arose in computer science. The early VO workers recognized that distributed processing would become as important as access to distributed data in the current era of large surveys; data volumes are rising beyond the point where all necessary data can be processed on a researcher's dekstop. The VO is thus evolving from being a unified set of web-sites for interactive querying of archives to being a data grid (i.e. a wide-area, distributed filing system) for local applications and finally into a computational grid for distributed processing. Regional VO projects focus on different stages of this evolution; AstroGrid is strongly focussed on the later stages of the evolution and has exceptional support for large-scale computation.

The VO movement is regulated (at least in technical matters) by the International Virtual Observatory Alliance. More information about the VO movement can be found on the IVOA web-site.

About this document

This "document" is a guide to the architecture of AstroGrid's software framework (it is not the user manual!) and is intended to inform technical readers who must work with that software. As such, it combines short, introductory essays with a large volume of detail generated from a UML model; that model in turn is largely generated from the code base. Readers with different background and with different goals will need to consider different parts of the mass of information, so there is no unique path through the material.  Some recommended paths are shown here.

For service providers

If you intend to provide a service in the VO by deploying AstroGrid components, then you should read the section on service organization; this has links to the installations guides for the components.  Next read the essays on workflow and job orientation, Common Execution Architecture, the resource registry and MySpace; these affect all service providers. To support deployment of data-processing services, read the description of how applications are coupled to the CEA. If you plan to deploy your own resource registry, then you should further study the corresponding use-case which supports the installation guide for the registry.

For end-users

This is not the user manual; look on AstroGrid's main site for pointers to that. To understand technically what AstroGrid does for you as a researcher you might read the essay and use cases about workflows, since all data processing in AstroGrid is done via workflows. You may also find it useful to read the essays on the registry and on MySpace.

For software developers inside AstroGrid

If you already know the architecture, you can ignore the introductory essays and use the reference material. The section on the Maven projects lists the artefacts produced by each Maven project and the compile-time dependencies. It also includes lists of the packages contained in each project. The section on the code-base presents the code as a unified tree which may sometimes be more useful than the fragmented view in the pages produced by Maven during the scheduled builds. The section on web applications lists the main interfaces for each sub-system, including the set of  web services; this information is not readily available from the documentation produced by Maven. There is also a section of the model containing design patterns.

For those with a non-HTML form of the document

This material isn't properly readable except as hypertext; there's too much detail to handle that way. Therefore, the "document" is issued as a web site. If you have a serialized copy (e.g. paper or PDF), then I suggest that you read only the introductory essays and refer to the on-line version for the UML details.
