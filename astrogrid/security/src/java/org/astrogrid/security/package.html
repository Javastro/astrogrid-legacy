Secure messaging and control of access to resources for web services and their clients.

'Secure messaging' means that messages to and from services are protected against tampering (e.g. by digital signature), or made confidential (e.g. by encryption) or made proof against misuse (e.g. by using security contexts to prevent an attacker replaying captured messages). 'Access control' means that the identity of the the sender of the message is checked and made known to the service and that use of a service is checked against recorded policy.

The classes are in two groups: SecurityGuard and its sub-classes; Axis message-handlers. Applications only make direct calls to the SecurityGuard variants; the other classes are called from within the framework.

A SecurityGuard is a Java bean whose properties represent credentials. The basic bean is simply an abstract way of passing credentials between objects inside one JVM; it has no methods for manipulating or using the credentials.

The AxisClientSecurityGuard extends SecurityGuard with code for passing credentials to an Axis handler. It depends internally on APIs from Axis and WSS4J. A client should use the static method getEngineConfiguration to configure an instance of org.apache.axis.client.Service with the apropriate handlers. The client should then call configureStub on the security guard to pass the credentials into the handler.

AxisClientCredentialHandler is the handler that signs outgoing messages. It is the user of the grid credentials.

The AxisServiceSecurityGuard extends SecurityGuard with a factory method getInstanceFromContext that constructs a guard with credentials retrieved from the authentication handler. A service may use this to retrieve the authenticated identity of the caller.

AxisServiceCredentialHandler is the handler for the services. It checks signatures on incoming messages. Services should declare handlers of this class in their request-processing chains.

The other classes in the package should not be called directly by client code.
